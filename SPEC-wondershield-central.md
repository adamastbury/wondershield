# WonderShield Central — Technical Specification

**Version:** 1.1 Draft
**Date:** 23 March 2026
**Author:** Winston Brambles (on behalf of Adam Astbury, Wonder Media Ltd)

---

## 1. Overview

WonderShield Central is a lightweight admin dashboard hosted on **Cloudflare Pages** with a **Cloudflare D1** (serverless SQLite) database. It provides a single pane of glass for monitoring all WordPress sites running the WonderShield security plugin (~30 domains), and enables one-click plugin updates across every site simultaneously.

### Goals

1. **Aggregated stats** — View security events (blocks, login attempts, probes, bad agents) across all domains from one dashboard.
2. **Per-site drill-down** — Click into any domain to see its individual stats and status.
3. **One-click mass update** — Push a WonderShield plugin update to all registered sites instantly, without logging into each WordPress admin.
4. **Real-time attack visibility** — See blocking events as they happen, not just on page load.
5. **Minimal server impact** — The reporting mechanism must be lightweight enough to run on shared hosting without affecting site performance.

### Non-Goals (v1)

- Multi-user auth / team access (just Adam for now).
- Managing WonderShield settings remotely (e.g. changing thresholds).
- Historical trend graphs or long-term analytics (keep it simple).

### Why This Architecture

- **Cloudflare Pages + Functions** = zero-ops hosting, global edge, free tier handles this scale easily.
- **D1** = serverless SQLite, no connection management, perfect for ~30 sites sending lightweight payloads.
- **Push model for stats** = the plugin sends data to the API, so the dashboard always has fresh data without needing to reach into each WordPress site (which would be slow and require exposing WordPress APIs).
- **Poll model for updates** = the plugin already polls GitHub every 5 minutes for updates. We piggyback on this existing mechanism to also check for a "force update" flag, keeping the update flow simple and secure (no inbound webhooks needed on WordPress sites).

## 3. Data Model (Cloudflare D1)

### 3.1 `sites` table

Registered WonderShield installations. Each site registers itself on first heartbeat.

```sql
CREATE TABLE sites (
    id              TEXT PRIMARY KEY,          -- UUID generated by the plugin on first registration
    domain          TEXT NOT NULL UNIQUE,      -- e.g. "example.co.uk"
    site_url        TEXT NOT NULL,             -- Full URL, e.g. "https://example.co.uk"
    site_name       TEXT,                      -- WordPress site title (from get_bloginfo)
    plugin_version  TEXT NOT NULL,             -- Currently installed WonderShield version
    wp_version      TEXT,                      -- WordPress version
    php_version     TEXT,                      -- PHP version
    last_heartbeat  TEXT NOT NULL,             -- ISO 8601 timestamp of last heartbeat
    registered_at   TEXT NOT NULL DEFAULT (datetime('now')),
    api_key         TEXT NOT NULL              -- Per-site API key for authenticating requests
);
```

### 3.2 `events` table

Security events pushed from each site in real time.

```sql
CREATE TABLE events (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    site_id     TEXT NOT NULL REFERENCES sites(id),
    event_type  TEXT NOT NULL,       -- 'blocked', 'login_failed', 'probe_blocked', 'bad_agent_blocked', 'xmlrpc_blocked', 'enum_blocked'
    ip          TEXT NOT NULL,
    path        TEXT,
    user_agent  TEXT,
    created_at  TEXT NOT NULL,       -- Timestamp from the originating site (UTC)
    received_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX idx_events_site ON events(site_id, created_at);
CREATE INDEX idx_events_type ON events(event_type, created_at);
```

### 3.3 `stats_snapshots` table

Periodic summary stats from each heartbeat (lightweight aggregate data).

```sql
CREATE TABLE stats_snapshots (
    id                  INTEGER PRIMARY KEY AUTOINCREMENT,
    site_id             TEXT NOT NULL REFERENCES sites(id),
    blocked_24h         INTEGER DEFAULT 0,
    blocked_7d          INTEGER DEFAULT 0,
    blocked_30d         INTEGER DEFAULT 0,
    attempts_24h        INTEGER DEFAULT 0,
    xmlrpc_blocked      INTEGER DEFAULT 0,
    probes_blocked      INTEGER DEFAULT 0,
    total_events        INTEGER DEFAULT 0,
    active_blocks       INTEGER DEFAULT 0,
    snapshot_at         TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX idx_snapshots_site ON stats_snapshots(site_id, snapshot_at);
```

### 3.4 `update_commands` table

Tracks force-update requests and per-site status.

```sql
CREATE TABLE update_commands (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    target_version  TEXT NOT NULL,              -- The version to update to
    issued_at       TEXT NOT NULL DEFAULT (datetime('now')),
    issued_by       TEXT DEFAULT 'admin'
);

CREATE TABLE update_status (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    command_id      INTEGER NOT NULL REFERENCES update_commands(id),
    site_id         TEXT NOT NULL REFERENCES sites(id),
    status          TEXT NOT NULL DEFAULT 'pending',  -- 'pending', 'updating', 'success', 'failed'
    old_version     TEXT,
    new_version     TEXT,
    error_message   TEXT,
    updated_at      TEXT NOT NULL DEFAULT (datetime('now')),
    UNIQUE(command_id, site_id)
);
```

## 4. API Endpoints (Cloudflare Pages Functions)

All endpoints live under `/api/` and are implemented as Cloudflare Pages Functions (in `functions/api/`).

### 4.1 Authentication

Three auth mechanisms:

1. **Site registration** (first heartbeat only): A shared `REGISTRATION_SECRET` is hardcoded into the plugin and stored as a Cloudflare env var. On first heartbeat, the plugin sends `Authorization: Bearer <REGISTRATION_SECRET>` along with its `site_id`. The API validates this, generates a unique per-site API key, stores it in D1, and returns it in the response. The plugin saves this key to `wp_options` as `ws_central_api_key`. All subsequent requests use the per-site key. If the registration secret is ever compromised, rotate it in Cloudflare and push a plugin update.

2. **Plugin → API** (heartbeats, events, update checks): Each site has a unique `api_key` stored in D1 and in the WordPress `wp_options` table. Sent as `Authorization: Bearer <site_api_key>`. If a 401 is received, the plugin should log a warning to the WonderShield admin page but continue operating normally — never retry auth in a loop.

3. **Dashboard → API** (admin actions): A single admin secret stored as a Cloudflare environment variable (`ADMIN_SECRET`). Sent as `Authorization: Bearer <admin_secret>`. Additionally, use **Cloudflare Access** in front of the entire dashboard (recommended — it's free for one user and gives you SSO via email OTP). Admin POST endpoints should also validate a `X-CSRF-Token` header that matches a token returned by `GET /api/auth/csrf`.

### 4.2 Plugin Endpoints (called by WonderShield plugin)

#### `POST /api/heartbeat`

Called by each site on a lightweight WP cron schedule (every 15 minutes). Registers/updates the site and sends summary stats.

**Request body:**
```json
{
    "site_id": "uuid-here",
    "domain": "example.co.uk",
    "site_url": "https://example.co.uk",
    "site_name": "Example Site",
    "plugin_version": "1.4.2",
    "wp_version": "6.7",
    "php_version": "8.2.0",
    "stats": {
        "blocked_24h": 12,
        "blocked_7d": 45,
        "blocked_30d": 120,
        "attempts_24h": 200,
        "xmlrpc_blocked": 50,
        "probes_blocked": 80,
        "total_events": 500,
        "active_blocks": 3
    }
}
```

**Response (existing site):**
```json
{
    "ok": true,
    "force_update": false,
    "target_version": null,
    "command_id": null
}
```

**Response (first registration — only when using REGISTRATION_SECRET):**
```json
{
    "ok": true,
    "api_key": "ws_sk_generated_unique_key_here",
    "force_update": false,
    "target_version": null,
    "command_id": null
}
```

The plugin must store the returned `api_key` in `wp_options` immediately and use it for all subsequent requests.

If `force_update` is `true` and `target_version` is set, the plugin should schedule a self-update via a one-off WP cron event (see Section 6).

**Logic:**
1. Validate `Authorization` header. If it matches `REGISTRATION_SECRET` and `site_id` is not in D1, generate a per-site API key, insert the site, and return the key. If it matches a stored per-site `api_key`, proceed. Otherwise return 401.
2. Upsert into `sites` table (update `last_heartbeat`, `plugin_version`, etc.).
3. Insert into `stats_snapshots`.
4. Check `update_commands` for the latest pending command where this site's status is `pending` → return `force_update: true` with the `command_id` and `target_version`.

**Error responses:**
- `401 Unauthorized` — invalid or missing API key. Plugin should log a warning and retry on next heartbeat cycle, not in a tight loop.
- `400 Bad Request` — malformed body. Plugin should log the error.
- `500 Internal Server Error` — Central is having issues. Plugin should silently retry on next cycle.

#### `POST /api/event`

Called by the plugin in real-time whenever a significant security event occurs (IP blocked, login failed, etc.). This is the "phone home on attack" mechanism.

**Request body:**
```json
{
    "site_id": "uuid-here",
    "events": [
        {
            "event_type": "blocked",
            "ip": "192.168.1.100",
            "path": "/wp-login.php",
            "user_agent": "Mozilla/5.0...",
            "created_at": "2026-03-23T14:30:00Z"
        }
    ]
}
```

**Response:**
```json
{
    "ok": true,
    "received": 1
}
```

Events are batched — the plugin queues events in a transient and flushes them every 60 seconds (or immediately if there are 10+ events, indicating an active attack). This prevents hammering the API on every single blocked request during a brute force attack.

#### `GET /api/sites/:site_id/check`

Lightweight poll endpoint the plugin calls alongside its existing GitHub release check (every 5 minutes). Returns only update instructions.

**Response:**
```json
{
    "force_update": false,
    "target_version": null,
    "command_id": null
}
```

Or when an update is pending:
```json
{
    "force_update": true,
    "target_version": "1.5.0",
    "command_id": 42
}
```

## 5. Plugin Modifications (WonderShield WordPress Plugin)

The existing WonderShield plugin needs the following additions. All changes should be backwards-compatible — if the central API is unreachable, the plugin continues to function exactly as it does today.

### 5.1 New Constants

Add to `wondershield.php`:

```php
define('WS_CENTRAL_API',       'https://shield.wondermedia.co.uk/api');  // Cloudflare Pages URL
define('WS_HEARTBEAT_INTERVAL', 15 * MINUTE_IN_SECONDS);                // 15 minutes
define('WS_EVENT_FLUSH_INTERVAL', 60);                                   // Seconds between event flushes
define('WS_EVENT_FLUSH_THRESHOLD', 10);                                  // Immediate flush if queue hits this
```

### 5.2 New File: `includes/central.php`

This file handles all communication with WonderShield Central. It should be required in `wondershield.php` after the existing includes.

#### Site Registration & API Key

On activation (or first heartbeat), generate a UUID (`wp_generate_uuid4()`) and store it as `ws_central_site_id` in `wp_options`. The API key is returned by the central API on first registration and stored as `ws_central_api_key` in `wp_options`.

#### Heartbeat (WP Cron)

Register a custom cron interval of 15 minutes. On each tick:

1. Gather stats using the existing `ws_get_stats()` function.
2. POST to `/api/heartbeat` with site info and stats.
3. If the response contains `force_update: true`, trigger the self-update mechanism (Section 6).
4. Use `wp_remote_post()` with a 5-second timeout to avoid blocking.
5. Wrap everything in a try/catch — if the API is down, silently fail and retry next tick.

#### Event Reporting

Hook into the existing `ws_log()` function. After logging to the local database, also queue the event for reporting to Central:

1. Maintain an event queue in a WordPress transient (`ws_central_event_queue`).
2. On each security event, append to the queue.
3. On `shutdown` hook, check if the queue has events and either:
   - Flush immediately if queue size >= `WS_EVENT_FLUSH_THRESHOLD` (active attack).
   - Flush if the last flush was > `WS_EVENT_FLUSH_INTERVAL` seconds ago.
   - Otherwise, leave for next request.
4. Only queue "interesting" event types: `blocked`, `login_failed`, `probe_blocked`, `bad_agent_blocked`, `xmlrpc_blocked`. Skip `attempt` and `blocked_hit` to keep volume manageable.

#### Update Check Piggyback

Modify `class-updater.php` to also call `/api/sites/:id/check` when it fetches the GitHub release. This adds negligible overhead (one extra HTTP request every 5 minutes) and means force-update commands are picked up within 5 minutes.

### 5.3 Performance Considerations

- **No synchronous API calls on user-facing requests.** The heartbeat runs on WP cron (async). Event reporting happens on the `shutdown` hook (after the response is sent to the visitor). The update check runs in the background transient refresh.
- **Graceful degradation.** If Central is unreachable, the plugin works identically to today. All API calls have short timeouts (5s) and silently fail.
- **Minimal data.** Heartbeats are a single small JSON POST every 15 minutes. Event flushes are batched. No sensitive data is transmitted (no usernames, passwords, or content — just IPs, event types, and paths).

## 6. Force Update Mechanism

This is the trickiest part. Here's how it works end-to-end:

### Flow

1. **Admin clicks "Update All"** on the Central dashboard.
2. Dashboard calls `POST /api/update/push` with the target version.
3. Central creates `update_commands` and `update_status` rows (one per site, status = `pending`).
4. Each WonderShield plugin picks up the command on its next check cycle (within 5 minutes max, typically sooner since heartbeats run every 15 mins and update checks every 5 mins).
5. Plugin sees `force_update: true` → triggers self-update.
6. Plugin reports back success/failure → Central updates `update_status`.
7. Dashboard polls `GET /api/update/status` to show live progress.

### Plugin Self-Update Implementation

**Critical: The update must NOT run synchronously during a web request.** When the plugin receives `force_update: true` from a heartbeat or check response, it should schedule a one-off WP cron event to perform the update asynchronously:

```php
// On receiving force_update: true
function ws_schedule_force_update($target_version, $command_id) {
    // Store the command details in a transient
    set_transient('ws_pending_force_update', [
        'target_version' => $target_version,
        'command_id'     => $command_id,
        'old_version'    => WS_VERSION,
    ], HOUR_IN_SECONDS);

    // Report "updating" status immediately
    ws_report_update_status($command_id, 'updating', WS_VERSION, null);

    // Schedule the actual update for the next available cron tick
    if (!wp_next_scheduled('ws_force_update_cron')) {
        wp_schedule_single_event(time(), 'ws_force_update_cron');
    }

    // Spawn cron immediately so it runs ASAP rather than waiting for a visitor
    spawn_cron();
}

// The actual update runs in WP cron context (async, no HTTP timeout risk)
add_action('ws_force_update_cron', 'ws_execute_force_update');
function ws_execute_force_update() {
    $pending = get_transient('ws_pending_force_update');
    if (!$pending) return;

    delete_transient('ws_pending_force_update');
    $command_id  = $pending['command_id'];
    $old_version = $pending['old_version'];

    try {
        require_once ABSPATH . 'wp-admin/includes/class-wp-upgrader.php';
        require_once ABSPATH . 'wp-admin/includes/plugin.php';

        // Clear the update transient cache to force a fresh check
        delete_site_transient('update_plugins');
        wp_update_plugins();

        // Run the upgrade
        $skin = new WP_Ajax_Upgrader_Skin();
        $upgrader = new Plugin_Upgrader($skin);
        $result = $upgrader->upgrade('wondershield/wondershield.php');

        // Check if it worked
        $new_version = get_plugin_data(WP_PLUGIN_DIR . '/wondershield/wondershield.php')['Version'];

        if (is_wp_error($result)) {
            ws_report_update_status($command_id, 'failed', $old_version, null, $result->get_error_message());
        } elseif (version_compare($new_version, $old_version, '>')) {
            ws_report_update_status($command_id, 'success', $old_version, $new_version);
        } else {
            ws_report_update_status($command_id, 'failed', $old_version, null, 'Version unchanged after upgrade attempt');
        }
    } catch (Exception $e) {
        ws_report_update_status($command_id, 'failed', $old_version, null, $e->getMessage());
    }
}
```

**Report-back retry logic:** If the POST to `/api/update/report` fails (network error), store the report payload in a transient and retry on the next heartbeat. The heartbeat can include an `update_report` field as a fallback delivery mechanism. Central should also treat any heartbeat from a site with a newer `plugin_version` than the pending command's `old_version` as an implicit success.

### Report Back

After the update (success or failure), the plugin calls:

```
POST /api/update/report
{
    "site_id": "uuid",
    "command_id": 42,
    "status": "success",         // or "failed"
    "old_version": "1.4.2",
    "new_version": "1.5.0",
    "error_message": null        // or error string
}
```

## 7. Dashboard UI (Cloudflare Pages SPA)

### 7.1 Tech Stack

- **Framework:** Plain HTML/JS with Alpine.js or Svelte (keep it lightweight, no need for React overhead for a single-user admin panel).
- **Styling:** Tailwind CSS.
- **Hosting:** Cloudflare Pages (static assets + Functions).
- **Auth:** Cloudflare Access (email OTP to adam@wondermedia.co.uk) — simplest option, one-user friendly, and adds authentication at the edge before any request reaches the app.

### 7.2 Pages

#### Dashboard (Home)

The main overview. Shows:

- **Top bar:** Total sites, total blocks (24h), total login attempts (24h), sites needing update.
- **Site cards/table:** Each site as a row or card showing domain, plugin version, last heartbeat (with online/offline indicator), key stats (blocks 24h, attempts 24h, active blocks).
- **Status indicators:**
  - 🟢 Green dot = heartbeat within 20 mins
  - 🟡 Yellow dot = heartbeat 20–60 mins ago
  - 🔴 Red dot = heartbeat >60 mins ago (site may be down or plugin deactivated)
- **Sorting:** By domain, by blocks (most attacked first), by version (outdated first), by status.
- **"Update All" button** at the top — only visible when sites are running an older version than the latest GitHub release. Shows target version.

#### Site Detail

Click a site to see:

- Full stats (all the numbers from `ws_get_stats()`).
- Recent events feed (last 100 events from `events` table).
- Current active blocks.
- Plugin version and environment info (WP version, PHP version).
- "Force Update" button for single-site update.

#### Update Progress

Shown after clicking "Update All" or "Force Update." Live-updating progress view as described in Section 6.

### 7.3 Graceful Loading

Per Adam's request — the dashboard should load instantly with a skeleton/placeholder UI, then populate site data progressively as it arrives. This is straightforward since all data is already in D1 (pushed by plugins), so it's just a single database query, not 30 outbound requests. Should load in <200ms.

A "Last updated" timestamp on each site card shows when the data was last refreshed (= last heartbeat time).

## 9. Security Considerations

1. **API keys per site.** Each WonderShield installation authenticates with a unique key. If a site is compromised, you can revoke its key without affecting others.

2. **Admin auth via Cloudflare Access.** The dashboard is behind Cloudflare Access, so nobody can even see the login page without a valid email OTP. The admin API endpoints additionally check the `ADMIN_SECRET` header.

3. **No inbound connections to WordPress.** The plugin only makes outbound POST/GET requests. No webhooks, no exposed endpoints on the WordPress side. This means there's zero additional attack surface on the WordPress sites.

4. **No sensitive data in transit.** The plugin sends IPs, event types, paths, and user agents. No credentials, no content, no user data.

5. **HTTPS everywhere.** Cloudflare Pages is HTTPS by default.

6. **Rate limiting.** Add Cloudflare rate limiting rules to the API endpoints to prevent abuse (though with per-site API keys, this is already fairly locked down).

## 14. Data Retention & Cleanup

D1 doesn't have built-in TTLs, so implement a **Cloudflare Cron Trigger** (defined in `wrangler.toml`) that runs daily and:

1. **`events` table:** Delete rows older than 90 days. `DELETE FROM events WHERE received_at < datetime('now', '-90 days');`
2. **`stats_snapshots` table:** Keep only the most recent 7 days of snapshots at full resolution. For older data, keep one snapshot per day per site (the latest one). Delete everything older than 90 days. This prevents unbounded growth (~30 sites × 96 snapshots/day = 2,880 rows/day without cleanup).
3. **`update_commands` / `update_status`:** Keep for 90 days, then delete. Old update history has no operational value.
4. **`sites` table:** Never auto-delete. If a site hasn't heartbeated in 30+ days, flag it as `inactive` in the dashboard (but don't remove — the admin can manually remove decommissioned sites).

The cron trigger is shared with the monitoring system (Section 17). It runs every 5 minutes for offline/attack checks, and performs the data retention cleanup when the current hour is 3am UTC. See Section 17 for the `wrangler.toml` config.

## 15. Event Batching Limits

To prevent oversized payloads during heavy attacks:

- **Max batch size:** 50 events per `POST /api/event` request. If the queue exceeds 50, send multiple batches.
- **Max queue size in transient:** 200 events. If the queue hits 200 without being flushed (e.g. cron isn't running), drop the oldest events. Better to lose some event data than to build up unbounded memory usage.
- **Max request body size:** The API should reject any request body over 100KB with a 413 status code.

## 16. API Key Management

- **Rotation:** Add a `POST /api/sites/:id/rotate-key` admin endpoint that generates a new API key, stores it in D1, and returns it. The admin then needs to manually update the site's `wp_options` (or the plugin can expose a settings field for the API key). For v2, consider a key rotation protocol where Central pushes a new key via the heartbeat response.
- **Revocation:** Delete the site's row from `sites` (or set `api_key` to null). The plugin will start receiving 401s and log warnings, but will continue to protect the site locally as normal.

## 17. Pushover Notifications

Central sends push notifications via **Pushover** (https://pushover.net) for critical events. Pushover is a simple REST API — no SDKs needed, just a POST to `https://api.pushover.net/1/messages.json`.

### Notification Triggers

These are checked by the **Cloudflare Cron Trigger** (runs every 5 minutes alongside the daily cleanup — add `*/5 * * * *` to the crons array) and also inline during event processing:

1. **Site offline** — A site's `last_heartbeat` is older than 30 minutes. Send once, then don't repeat until the site comes back online and goes offline again. Track notification state in a `site_alerts` table (see below).
2. **Mass attack detected** — A single site receives 50+ block events within a 5-minute window. Check this during `POST /api/event` processing. Throttle to one notification per site per 30 minutes.
3. **Update failed** — A site reports `status: failed` during a force-update operation. Send immediately on receiving the failure report.
4. **Update complete** — All sites in an update command have reported back (all success, or mix of success/failed). Send a summary notification.

### Pushover Message Format

```
POST https://api.pushover.net/1/messages.json
Content-Type: application/x-www-form-urlencoded

token=<PUSHOVER_APP_TOKEN>
&user=<PUSHOVER_USER_KEY>
&title=WonderShield: Site Offline
&message=example.co.uk hasn't checked in for 35 minutes
&priority=1
&sound=siren
&url=https://shield.wondermedia.co.uk/sites/uuid-here
&url_title=View in Central
```

Priority levels:
- `priority=1` (high) — site offline, update failed
- `priority=0` (normal) — mass attack detected, update complete

### `site_alerts` Table

Tracks notification state to prevent spamming:

```sql
CREATE TABLE site_alerts (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    site_id     TEXT NOT NULL REFERENCES sites(id),
    alert_type  TEXT NOT NULL,          -- 'offline', 'attack', 'update_failed'
    last_sent   TEXT NOT NULL,          -- ISO 8601 timestamp
    resolved    INTEGER DEFAULT 0,     -- 1 when the condition clears
    UNIQUE(site_id, alert_type)
);
```

**Offline detection logic:** The cron queries `SELECT * FROM sites WHERE last_heartbeat < datetime('now', '-30 minutes')`. For each result, check `site_alerts` — if no unresolved `offline` alert exists for that site, send a Pushover notification and insert/update the alert row. When a heartbeat arrives for a site with an unresolved offline alert, set `resolved = 1`.

### Cron Schedule Update

Update `wrangler.toml` to run both the cleanup and the monitoring:

```toml
[triggers]
crons = ["*/5 * * * *"]  # Every 5 mins: offline checks, attack detection. Daily cleanup runs within the same handler (check hour == 3).
```

The handler function checks the current hour — if it's 3am UTC, also run the data retention cleanup from Section 14.

## 18. Frontend Framework Decision

**Recommendation: Svelte** (compiled to vanilla JS, deployed as static assets to Cloudflare Pages).

Rationale:
- Compiles down to tiny, fast vanilla JS bundles — no runtime framework shipped to the browser.
- Reactivity is built-in (perfect for the live-updating progress view during mass updates).
- Far less boilerplate than React for a dashboard this size.
- Svelte is fully static-compatible — `npm run build` produces HTML/JS/CSS that drops straight into Cloudflare Pages with zero config.
- SvelteKit (Svelte's app framework) supports Cloudflare Pages adapter out of the box if we want SSR later, but for v1 a pure SPA build is fine.

The dashboard is small enough (~3 pages) that the choice barely matters, but Svelte will give the nicest developer experience with the smallest output.

Use **Tailwind CSS** for styling. Use **SvelteKit** with the `@sveltejs/adapter-cloudflare` adapter so the Pages Functions and SPA are in one unified project.
